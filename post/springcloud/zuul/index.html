Zuul

hystrix的两种策略模式

  1. 线程池模式
  2. 信号量模式

线程池

  • 优点：通过线程池分配线程进行处理，不需要同步等待，可以设置超时时间，超时则进行超时处理，释放线程资源
  • 缺点：需要浪费资源不断进行线程上下文的切换，不适用于对时效性要求很高的场景

信号量

  • 优点：通过获取信号量的方式，在一个线程内进行，所有的操作共享threadlocal中的数据项
  • 缺点：不能设置超时时间，同步等待。一旦服务假死，则阻塞所有信号量，服务器将不处理其他请求

Feign中的hystrix设置

hystrix默认模式thread

feign:
  hystrix:
    enabled: true #开启feign的hystrix
  okhttp:
    enabled: true
  httpclient:
    max-connections: 2000 # 默认值
    max-connections-per-route: 50 # 默认值

hystrix:
  threadpool:
    default:
      coreSize: 2500  #并发执行的最大线程数，默认10
  command:
    default:
      execution:
        timeout:
          enabled: true
        isolation:
          strategy: thread #这个配置只会在feign生效
          thread:
            timeoutInMilliseconds: 10000
          semaphore:
            maxConcurrentRequests: 2000

zuul中的hystrix设置

zuul的配置中重写了hystrix的一些配置，和feign中使用的不同。zuul中hystrix默认使用semaphore

zuul:
  host:
    socket-timeout-millis: 5000
    connect-timeout-millis: 5000
    FormBodyWrapperFilter:
      pre:
        disable: true
    max-total-connections: 500
    max-per-route-connections: 200
  semaphore:
    max-semaphores: 2000
  retryable: true
  ribbon-isolation-strategy: semaphore #可修改为线程池

使用流程

zuul作为网关服务进行鉴权和转发

  1. 编写自定义filter决定是否鉴权
  2. 不鉴权的直接进行zuul的下游filter，鉴权的则进行鉴权。请求鉴权服务使用的feign进行请求（踩坑）
  3. zuul的路由是采用了接入eureka，并加入db中的自定义转发规则，使用了linkedlist，先接入自定义，再接入eureka中的服务，这样实现灵活的route规则，不需要在配置文件写配置
  4. zuul中有了转发规则，可以根据配置转发到某个服务，采用ribbon进行负载均衡的转发。ribbon有几个默认的负载均衡规则，进行了重写加强控制力。（在lb中利用eureka的metadata实现了金丝雀功能）
  5. 写fallback的filter进行处理
  6. 利用prometheus+grafana进行监控

踩坑记录

  • 自定义zuulfilter进行鉴权功能，其中通过feignclient进行请求，需要本地线程变量中的serviceId。feingnclient请求也会如同一个新请求，进入zuul的fliter进行处理。

现象： feignClient转发请求时，由于hystrix采用的thread模式，requrest.context中通过threadlocal存储的信息全部丢失，

解决方案： 在请求参数中传递需要的信息，通过请求拦截器进行拦截，解析参数，放入zuul的RequestContext.getCurrentContext()，再将数据进行移除

import feign.RequestTemplate;

@Component
public class AuthInterceptor implements RequestInterceptor {
    
    @Override
    public void apply(RequestTemplate requestTemplate) {

    }
}
  • 后端服务出问题陷入假死，不能及时处理请求。当时zuul使用的是信号量模式，导致网关直接hang死，qps直接掉底，后端服务恢复之后，才释放资源自动恢复 解决方案： 将隔离模式改成线程池。 ps：未解决 改为线程池直接导致自定义lb无效，找不到server，原因是不共享threadlocal中的信息，lb需要的key获取不到
